# App 前端

## 概述

这个技能用于处理 App 端或 H5 端前端接入，重点关注登录上下文、组织切换、多端接口一致性和真机联调。

开始处理移动端公共组件、token、样式和多语言能力前，先读 `bluexii-components2` 技能。

核心原则：

1. 先确认 App 宿主怎样传 token 和组织上下文。
2. 同一业务的 App 端和 PC 端接口语义必须一致。
3. H5 里不要默认复用 PC 的上下文读取方式。

## 何时使用

- App、H5、混合容器接入微服务
- 同一业务有多端并存
- 组织切换、登录态、页面回流行为异常

## 核心模式

### 固定接入顺序

1. 先确认当前是 App、H5、混合容器还是独立浏览器场景
2. 再确认 token、组织、locale 由谁提供
3. 再确认哪些能力必须直接复用公共仓
4. 最后才决定本地页面如何适配

### 上下文来源

- 宿主注入参数
- WebView bridge
- 登录态缓存
- token 兜底

### 公共能力优先

- 多语言统一走 `@bluexii/shared-i18n`
- 主题、样式、格式化优先复用 `@bluexii/mobile-*` 和 `shared-*`
- 不要在业务仓平行实现本地语言切换入口或公共样式 token

### 宿主管理边界

- 先分清楚是谁控制 token、组织、locale
- 宿主接管 `locale/language` 后，本地切换入口必须失效或隐藏
- H5 里不要把 PC 壳层运行时假设原样搬过来
- 真机、容器、浏览器三种路径分开验证，不要混成一条结论

### 多端接口对齐

- 字段语义一致
- 状态枚举一致
- 组织隔离规则一致
- 动态标题、状态文案、后端返回展示文案不要再做二次翻译

## 常见错误

- 直接照搬 PC 端存储策略
- 只在前端适配，不看后端是否真支持多端
- 真机和浏览器行为不一致但没有拆开验证
- 业务仓自己维护一套语言切换入口
- 宿主管控后，页面仍保留第二套本地语言控制
