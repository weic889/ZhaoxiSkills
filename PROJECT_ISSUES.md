# 项目问题汇总

这个文件是本仓库的开发前必读入口。

AI 在开始开发、联调、重构、提测前，默认先读这个文件，再决定是否继续阅读 `skills/` 下的专项材料。

## 先读这个文件的原因

这个仓库里沉淀的不是通用教程，而是项目里反复踩过的真实问题。

如果跳过这些问题总结，最常见的结果不是“实现不了”，而是：

1. 页面能跑，但接入方式不符合宿主和公共仓约束
2. 前后端字段能通，但组织、路由、语言控制权不一致
3. 本地联调正常，到了基座、QA、UAT、Prod 就出问题
4. 业务仓自己写出第二套公共能力，后续越来越难维护

## 开发前固定顺序

1. 先判断当前任务属于哪一类：
   - `pc-frontend`
   - `app-frontend`
   - `enterprise-portal-frontend`
   - `enterprise-portal-backend`
   - `pcapp-backend`
2. 先确认这次问题属于公共层还是业务层
3. 先确认宿主、组织、token、locale、path 由谁控制
4. 先确认是否已有公共仓能力可直接复用
5. 再开始写代码

## 最容易反复出现的问题

### 1. 公共的没有修公共

典型表现：

- 业务仓平行实现壳层、分页、筛选区、语言切换入口
- 公共仓已有能力，但业务仓再写一套“差不多”的版本
- 某个问题其实会在多个项目重复出现，却只在单个业务仓打补丁

默认规则：

1. 公共的修公共，业务的修业务
2. 如果问题会在多个仓重复出现，优先判断能不能沉淀回公共层
3. 业务仓允许临时兜底，但不能默认长期保留分叉实现

### 2. 组织上下文不一致

典型表现：

- 页面显示组织 A，请求却还是组织 B
- 切组织后列表刷新了，详情、新增、编辑没刷新
- 前端请求头是对的，但后端查询过滤没跟上

默认规则：

1. 前端必须明确当前组织来源
2. 后端保存和查询都必须显式落组织范围字段
3. 列表、详情、编辑、导出共用同一套组织隔离语义

### 3. 多语言控制权冲突

典型表现：

- 宿主已经接管 `locale/language`，业务仓还保留第二套本地切换入口
- `QA / UAT / Prod` 里还能看到调试用语言切换按钮
- 后端已经返回动态标题或展示文案，前端又做二次翻译

默认规则：

1. 多语言统一走公共能力
2. 宿主接管语言后，本地切换入口必须隐藏或失效
3. 动态标题、菜单标题、状态展示文案如果后端已返回展示值，前端不要二次翻译

### 4. 宿主与子应用路由不同步

典型表现：

- 子应用内部切页了，外层 URL 不变
- 刷新后只能回首页，回不到真实内部页
- 点一次菜单出现多次 `navigate`、`replace`、`postMessage`

默认规则：

1. 宿主首次打开时必须把内部 `path` 回灌给子应用
2. 子应用切页后必须上报最新 `path`
3. 宿主 URL、标签缓存、刷新恢复要同步更新
4. 路由同步必须做语义去重和幂等保护
5. 路由决定菜单，不做菜单反向纠偏

### 5. 只看页面显示，不看运行契约

典型表现：

- 页面看起来没问题，但 token、baseURL、静态资源路径、分页请求不对
- 应用详情能返回，但不足以支撑宿主做路由恢复和入口控制
- 接口 200 了，但字段语义不一致

默认规则：

1. 不只验证 HTTP 200
2. 必看 token、组织、locale、resources、path 等运行时数据
3. 列表、详情、编辑、导出接口字段语义必须一致
4. 构建产物不能残留错误的根路径静态资源引用

## 各类项目的硬规则

### PC 前端

1. 先分清页面模式：列表页、详情页、表单页、双栏页、统计页、弹窗/抽屉页
2. 优先复用公共骨架，不在业务仓平行重写
3. 分页必须真实驱动请求
4. `QA / UAT / Prod` 下本地调试语言切换入口必须不可见
5. 跨域 `iframe` / 模块式 React 子应用必须验证宿主 URL、标签缓存、内部路由一致

详情见：
- `skills/pc-frontend/SKILL.md`

### App 前端

1. 先分清当前是 App、H5、混合容器还是浏览器
2. 先确认 token、组织、locale 由宿主还是本地控制
3. 真机、容器、浏览器三条链路分开验证
4. 不要把 PC 壳层运行时假设直接搬到 H5

详情见：
- `skills/app-frontend/SKILL.md`

### 企业门户前端

1. 门户是宿主，不只是跳板页
2. 宿主要统一下发 token、locale、resources、path
3. 子应用要支持 `path` 回灌、路由上报、刷新恢复
4. 同一路径同步要做幂等去重

详情见：
- `skills/enterprise-portal-frontend/SKILL.md`

### 企业门户后端

1. 门户后端返回的是运行契约，不只是展示信息
2. 应用详情、资源树、权限、组织范围必须语义一致
3. 要给宿主足够稳定的 `application` / `resource` / `path` 元数据
4. 动态菜单标题、应用标题、状态文案不要逼前端本地修补

详情见：
- `skills/enterprise-portal-backend/SKILL.md`

### PCApp 后端

1. 保存和查询都要显式处理组织隔离
2. 网关、配置、数据库、服务部署要当成同一条排障链
3. 列表、详情、编辑、导出接口字段语义要一致
4. 动态标题、状态文案、名称文案不要依赖前端补丁兜底

详情见：
- `skills/pcapp-backend/SKILL.md`

## 提测前阻断项

满足以下任一项，就不应该继续按“可提测/可发版”推进：

1. 宿主接管 `locale/language` 后，业务仓仍存在第二套可操作入口
2. `QA / UAT / Prod` 环境还能看到本地调试语言切换入口
3. 跨域 `iframe` / 模块式 React 子应用没有完成宿主路由同步验证
4. 刷新后回不到真实内部页
5. 列表分页看起来能点，但没有真实驱动请求
6. `dist/index.html` 仍残留 `src="/assets/...` 或 `href="/assets/...`
7. 后端接口之间字段语义、资源编码、组织范围语义不一致
